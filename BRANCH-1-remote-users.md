# Ветка 1. Удаленные пользователи через Cudy

## Цель

Сделать так, чтобы родственники и друзья могли установить простое VPN-приложение, подключиться к `195.170.35.108`, а их трафик дальше обрабатывался Cudy так же, как трафик клиентов локальной сети.

## Предпочтительная реализация

Начать с готового приложения:

- WireGuard App - самый простой и прозрачный вариант;
- AmneziaVPN - возможный вариант, если нужен AmneziaWG или более удобная оболочка.

Свое приложение делать только после проверки готовых клиентов. Оно потребуется только если импорт профилей, включение/выключение и обновление конфигов окажутся неудобными для обычных пользователей.

## Архитектура

```text
Удаленный телефон/ПК
  -> интернет
  -> 195.170.35.108:UDP_PORT
  -> Cudy OpenWrt
  -> входной VPN-интерфейс friends/wg_in
  -> PBR/nft rules
  -> awg1 / awg2 / будущие каналы / WAN
```

## Что надо выяснить

- На каком устройстве находится публичный IP `195.170.35.108`.
- Если Cudy за основным роутером, есть ли проброс UDP-порта на Cudy.
- Какой порт использовать для входного VPN.
- Нужен ли полный туннель для пользователей или только split tunnel.
- Нужно ли давать удаленным пользователям доступ к локальной сети или только к интернету через Cudy.

## План работ

1. Проверить внешний доступ:
   - текущий WAN IP;
   - проброс порта;
   - доступность UDP снаружи.

2. Создать отдельный входной VPN-интерфейс:
   - имя: `wg_in` или `friends`;
   - подсеть, например `10.77.0.0/24`;
   - отдельный порт, например `51830/udp`.

3. Создать peers:
   - один peer на одного человека;
   - отдельный private/public key;
   - отдельный IP;
   - при необходимости отдельный QR-код.

4. Настроить firewall:
   - разрешить входящий UDP-порт с WAN;
   - разрешить forwarding из зоны удаленных пользователей в нужные зоны;
   - включить masquerading там, где нужно.

5. Подключить удаленную VPN-сеть к PBR:
   - проверить, что трафик из `10.77.0.0/24` попадает в те же PBR-правила;
   - при необходимости добавить отдельные policies для source address.

6. Проверить одного тестового пользователя:
   - подключение;
   - DNS;
   - внешний IP;
   - Telegram/YouTube/ChatGPT;
   - отсутствие лишнего доступа к домашней LAN, если он не нужен.

## Результат ветки

- Рабочий шаблон клиента.
- Инструкция для пользователя.
- Набор команд для добавления/удаления peer.
- Понимание, нужен ли собственный клиент или достаточно WireGuard/AmneziaVPN.

## Первый следующий шаг

На Cudy/OpenWrt собрать диагностику внешнего входа:

```sh
ip -4 addr
ip -4 route
uci show firewall | grep -Ei 'wan|redirect|rule|518|udp'
```

С внешней сети проверить доступность выбранного UDP-порта после настройки проброса.

## Результат первичной проработки

Рекомендуемое начальное решение:

- входной сервер делать отдельным интерфейсом `wg_in` / `friends`;
- не смешивать его с исходящими туннелями `awg1` и `awg2`;
- первый клиент: обычное приложение WireGuard;
- подсеть входных клиентов: `10.77.0.0/24`;
- адрес Cudy на входном VPN: `10.77.0.1/24`;
- первый клиент: `10.77.0.2/32`;
- порт для теста: `51830/udp`;
- режим первого теста: full tunnel;
- доступ к домашней LAN на первом этапе закрыть;
- маршрутизация удаленных клиентов должна идти через те же PBR-правила, что и LAN.

Безопасная firewall-модель:

- открыть на WAN только `51830/udp` на сам Cudy;
- создать отдельную firewall-зону `friends`;
- разрешить `friends -> wan`;
- `friends -> lan` не разрешать до отдельного решения;
- не выдавать один общий peer всем пользователям.

Ключевые риски:

- публичный IP `195.170.35.108` может быть не на Cudy, тогда нужен UDP port forward;
- если у провайдера CG-NAT, входной VPN напрямую не заработает;
- цепочка `WireGuard client -> Cudy -> awg1/awg2` может потребовать MTU/MSS-настройки;
- full tunnel увеличит нагрузку на Cudy и исходящие туннели.

## Диагностика 2026-05-26

На Cudy публичного IP `195.170.35.108` нет в текущем тестовом режиме.

Фактическая тестовая схема сейчас:

```text
internet
  -> upstream router / public side
  -> 192.168.1.1
  -> Cudy WAN eth0: 192.168.1.174/24
  -> Cudy LAN br-lan: 192.168.8.1/24
```

Следствие: для входного WireGuard-сервера нужен проброс UDP-порта на upstream-роутере:

```text
195.170.35.108:51830/udp -> 192.168.1.174:51830/udp
```

На Cudy пока нет правила для `51830/udp`; порт выглядит свободным для новой ветки.

## Целевая схема после миграции

После настройки Cudy должен стать главным домашним роутером:

```text
internet / ISP / ONT
  -> Cudy WAN
  -> Cudy LAN: 192.168.1.1/24
```

В этой целевой схеме входной VPN для друзей должен приходить сразу на Cudy:

```text
remote client
  -> public IP:51830/udp
  -> Cudy wg_in/friends
  -> PBR
  -> awg1 / awg2 / WAN
```

Следствия для ветки:

- Тестовый port forward на `192.168.1.174` нужен только пока Cudy стоит за старым роутером.
- В финальной схеме правила firewall должны открывать `51830/udp` на WAN самого Cudy.
- Домашняя LAN после миграции будет `192.168.1.0/24`, а не текущая тестовая `192.168.8.0/24`.
- Если удаленным пользователям позже будет разрешен доступ к домашней сети, это будет доступ к `192.168.1.0/24`.

## Реализовано на Cudy 2026-05-26

Сначала был создан тестовый обычный WireGuard `wg_in`, затем по решению использовать менее заметный для DPI вариант он заменен на AmneziaWG.

Установлены пакеты для WireGuard/AmneziaWG-инфраструктуры:

- `kmod-wireguard`
- `wireguard-tools`
- `luci-proto-wireguard`
- `qrencode`

Активный входной сервер сейчас - AmneziaWG:

```text
interface: awg_in
address: 10.77.0.1/24
port: 51830/udp
mtu: 1280
first peer: test-client-awg, 10.77.0.2/32
```

Старый `wg_in` отключен, чтобы не конфликтовать за UDP `51830`.

AmneziaWG-клиентские файлы на роутере:

```text
/root/awg_clients/test-client-awg.conf
/root/awg_clients/test-client-awg.qr.txt
```

Локальная копия для тестового импорта:

```text
secrets/test-client-awg.conf
secrets/test-client-awg.qr.txt
secrets/test-client-awg.png
```

Бэкапы перед изменениями:

```text
/root/backup-before-wg-in-20260526-120803
/root/backup-before-awg-in-20260526-130042
```

Firewall:

- зона `friends` создана;
- `friends -> wan` разрешено;
- `friends -> awg1` разрешено;
- `friends -> awg2` разрешено;
- `friends -> lan` не разрешено;
- WAN input `51830/udp` разрешен;
- DNS и ping с `friends` на сам роутер разрешены.

Текущее ограничение:

- во временной тестовой схеме внешний доступ заработает только после проброса `195.170.35.108:51830/udp -> 192.168.1.174:51830/udp` на upstream-роутере;
- без проброса тестировать можно только изнутри сети или после финального перевода Cudy в главный роутер.

## Тест AmneziaVPN 2026-05-26

После импорта `test-client-awg.conf` в AmneziaVPN и подключения с мобильной сети:

- `awg show awg_in` увидел endpoint телефона;
- handshake был свежим;
- счетчики transfer росли;
- `conntrack` показал соединения:
  - source `10.77.0.2`;
  - destination внешние IP;
  - reply path через `10.8.1.10`, то есть `awg2`.

Вывод:

```text
Phone AmneziaVPN
  -> awg_in 10.77.0.1
  -> Cudy PBR
  -> awg2 10.8.1.10
  -> internet
```

Входной AmneziaWG для тестового клиента работает.

## Правка PBR для удаленных клиентов 2026-05-26

Симптом: AmneziaVPN на телефоне подключается к `test-client-awg.conf`, но сайты и приложения не получают нормальный доступ в интернет.

Проверка на Cudy показала, что:

- handshake и transfer на `awg_in` есть;
- `conntrack` видит соединения от `10.77.0.2`;
- ответы из интернета приходят;
- маршрутизация была смешанной: часть сессий через `awg2`, часть через обычный `wan`.

Для удаленных клиентов добавлена отдельная PBR-политика:

```text
name: Remote friends via awg2
src_addr: 10.77.0.0/24
interface: awg2
enabled: 1
```

Цель: весь IPv4-трафик входных AmneziaWG-клиентов сначала отправлять через текущий исходящий VPN-канал `awg2`, без смешивания с WAN. После reload PBR подтверждено правило:

```text
ip saddr 10.77.0.0/24 goto pbr_mark_0x030000 comment "Remote friends via awg2"
```

Бэкап: `/root/backup-before-friends-pbr-20260526-133603`.

Следующий тест: выключить/включить профиль AmneziaVPN на телефоне и открыть сайт, чтобы создать новые соединения уже после применения PBR.

## Переключение выхода удаленных клиентов 2026-05-26

Для сравнения выходов через `wan`, `awg1` и `awg2` на Cudy добавлены команды:

```text
friends-auto
friends-wan
friends-awg1
friends-awg2
friends-route {wan|awg1|awg2} [client-ip]
```

Команды меняют только PBR-политику для сети входных клиентов `10.77.0.0/24`, перезапускают PBR и сбрасывают conntrack для тестового клиента `10.77.0.2`, чтобы новые соединения пошли уже по выбранному пути.

Проверенный тестовый режим:

```text
10.77.0.0/24 -> wan
```

Бэкап перед тестами: `/root/backup-before-friends-route-test-20260526-134140`.

После успешной проверки прямого `wan` режим возвращен к штатной логике PBR:

```text
10.77.0.0/24 -> auto/PBR destination rules
```

Команда возврата:

```text
friends-auto
```

Смысл режима: нет принудительного правила по source `10.77.0.0/24`; трафик удаленных клиентов обрабатывается теми же destination-правилами, что и LAN. IP из активного списка идут через текущий `awg`, остальные напрямую через `wan`.

Бэкап перед возвратом: `/root/backup-before-friends-auto-20260526-135104`.

## Исключение vseinstrumenti.ru 2026-05-26

На удаленном AmneziaWG-клиенте сайт `https://www.vseinstrumenti.ru/` определял VPN.

Проверка показала:

- `www.vseinstrumenti.ru` резолвится в `185.169.155.85`;
- адрес входит в nft-set `pbr_awg2_4_dst_ip_user` как `185.169.155.0/24`;
- поэтому сайт шел через исходящий `awg2`, а не напрямую через `wan`.

Добавлена PBR-политика-исключение:

```text
Force vseinstrumenti via wan
interface: wan
dest_addr: 185.169.155.0/24
```

Правило применяется до большого списка `awg2`, поэтому должно перебивать его:

```text
ip daddr 185.169.155.0/24 goto pbr_mark_0x010000 comment "Force vseinstrumenti via wan"
ip daddr @pbr_awg2_4_dst_ip_user goto pbr_mark_0x030000
```

Бэкап: `/root/backup-before-vseinstrumenti-wan-20260526-141308`.

## Управление индивидуальными клиентами 2026-05-26

Для выдачи доступа друзьям/родственникам используется модель:

```text
1 человек = 1 AmneziaWG peer = 1 IP в 10.77.0.0/24 = 1 индивидуальный conf
```

Это нужно, чтобы:

- видеть, кто подключен;
- смотреть handshake и счетчики RX/TX по каждому peer;
- отзывать доступ одному человеку, не меняя конфиги остальных.

Команды на Cudy:

```text
friend-list
friend-add NAME [endpoint:port]
friend-show NAME
friend-qr NAME
friend-conf NAME
friend-revoke NAME
```

Пример:

```text
friend-add ivan
friend-qr ivan
friend-list
friend-revoke ivan
```

`friend-add` автоматически:

- выбирает свободный IP `10.77.0.x`;
- генерирует private/public key и preshared key;
- добавляет peer в `network` как `amneziawg_awg_in`;
- применяет peer live через `awg set`, без перезапуска всех клиентов;
- создает `/root/awg_clients/NAME-awg.conf`;
- создает `/root/awg_clients/NAME-awg.qr.txt`.

`friend-revoke`:

- удаляет peer из UCI;
- удаляет peer из live `awg_in`;
- переносит старый conf/QR в `/root/awg_clients/revoked/`;
- не отзывает доступ у других пользователей.

## Статистика и QR 2026-05-26

В `friend-list` поля статистики переименованы, чтобы не путать направление:

```text
from_peer_bytes  - сколько Cudy получил от клиента;
to_peer_bytes    - сколько Cudy отправил клиенту.
```

То есть `to_peer_bytes` - это примерно “сколько передали другу” через входной туннель. Для просмотра сайтов и видео это обычно основной растущий счетчик. Это не биллинг до байта: счетчики берутся из `awg show dump` и включают служебные накладные расходы туннеля.

QR для AmneziaVPN:

- наш QR - это raw QR от `.conf`;
- AmneziaVPN не принимает QR для `AmneziaWG native format`;
- официальный способ для нашего сценария - передавать индивидуальный `.conf` и импортировать его как файл;
- “мерцающий” QR в оригинальной Amnezia - это share-механизм самой AmneziaVPN (`vpn://...` / guest access), а не обычный `.conf` QR.

Следствие: `friend-qr` оставлен как вспомогательная команда, но для друзей основной путь:

```text
friend-add NAME
friend-conf NAME
```

## Исправление friend-add 2026-05-26

При попытке создать `DC_via_Cudy` скрипт ошибочно создал второго peer с именем `test-client-awg`. Причина: в POSIX shell переменные функций глобальные по умолчанию, и функция `peer_index_by_name` перезаписала переменную `name`.

Исправлено:

- переменные функции поиска peer переименованы, чтобы не затрагивать имя создаваемого клиента;
- новый формат файлов: `NAME-awg.conf` и `NAME-awg.qr.txt`;
- ошибочно созданный peer с IP `10.77.0.3/32` переименован в `DC_via_Cudy`;
- его конфиг теперь:

```text
/root/awg_clients/DC_via_Cudy-awg.conf
```

Контроль:

```text
friend-add DC_via_Cudy
-> Peer already exists: DC_via_Cudy

friend-list
-> test-client-awg 10.77.0.2
-> DC_via_Cudy     10.77.0.3
```
